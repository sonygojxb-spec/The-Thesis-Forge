# Multi-stage humanization pipeline for AI detection bypass

The original single-file LLM-wrapper app has been rebuilt into a five-stage pipeline: deterministic structural variation (spaCy), lexical substitution (WordNet + wordfreq), two-pass LLM rewrite (streaming against freemodel.dev), perplexity variance injection, and a post-processing fingerprint scrubber. A new Streamlit frontend exposes per-stage toggles, an intensity slider (1–5), before/after analytics, and a download button. The modular `humanizer/` package is well-separated from the UI layer.

**Watch for:** The streaming display discards Pass 1 output when Pass 2 begins, so the user sees only the second pass arrive — but the final displayed text depends on whether post-processing changes anything, which may silently replace what was streamed (**confirmed**). The pipeline instantiates the LLM rewriter with `lexical_aggression` instead of a dedicated LLM aggression key, meaning intensity calibration for the LLM stage is wrong at every level (**confirmed**). NLP stages rely on `random` without seeding, making results unreproducible across runs and making debugging impossible (**confirmed**). The hardcoded API key placeholder is shipped in committed source with no `.env` or secrets-management guidance (**confirmed**).

## High-level view

The pipeline orchestrator (`pipeline.py`) chains five stages, with each stage receiving a single `aggression` float derived from the intensity profile in `config.py`. The design cleanly separates deterministic NLP transforms from the LLM pass, letting users disable the expensive network call while still getting structural and lexical changes. However, the aggression parameter forwarded to the LLM stage is pulled from the wrong config key (`lexical_aggression`), so LLM temperature scaling doesn't track the intended intensity curve.

The streaming path in `process_stream` is a producer that yields LLM chunks live, then emits a sentinel `__FINAL__` followed by the post-processed text if it differs from what was streamed. The frontend renders chunks as they arrive, then swaps the full display to the final text. This means the user watches streaming output that may be entirely replaced once perplexity/postprocess stages run — a confusing UX when post-processing applies aggressive changes.

The NLP stages (structural, lexical, perplexity, postprocess) are all non-deterministic — they use `random.random()` and `random.choice()` without accepting a seed. This means identical inputs at identical settings yield different outputs, which makes QA and user trust difficult. For an academic tool where a user may want to compare runs, this is a practical gap.

The lexical stage's `_replace_target_words` method iterates over regex matches while mutating the underlying string, which corrupts match positions after the first replacement per word. Only the `break` statement prevents this from causing obvious damage, but it also limits the stage to one replacement per target word per invocation regardless of aggression.

The AI risk scoring heuristic in `text_analysis.py` is entirely self-referential — it measures properties the pipeline itself manipulates (transition density, TTR, sentence variance) rather than anything validated against actual Turnitin signals. The score will always drop after processing, giving the user false confidence that has no empirical relationship to real detection outcomes.

## Issues

<details>
<summary>Issues (8)</summary>

1. **LLM aggression uses wrong config key** — `pipeline.py` passes `stage_config.get("lexical_aggression")` to `LLMRewriter` instead of a dedicated LLM aggression value. Add an `llm_aggression` key to each intensity profile and use it.
2. **Streaming display discards Pass 1** — `process_stream` resets `llm_chunks` when it encounters the `---PASS2---` sentinel, so the frontend only displays Pass 2 chunks. If Pass 2 fails or returns empty, the user sees nothing despite Pass 1 succeeding.
3. **Post-processing silent replacement** — The `__FINAL__` sentinel swaps the entire displayed output after streaming completes. Indicate to the user that post-processing is altering the text, or show a diff.
4. **Non-deterministic NLP stages** — All random-based stages lack a configurable seed. Add a `seed` parameter to the pipeline for reproducibility.
5. **Lexical regex mutation during iteration** — `_replace_target_words` mutates `text` while iterating regex matches on the original string, corrupting positions. Collect replacements first, apply in reverse order.
6. **AI risk score is circular** — The heuristic only measures features the pipeline targets. It will always report improvement. Rename to "internal confidence" or add a disclaimer.
7. **API key in committed source** — `config.py` contains a hardcoded placeholder with no `.env` or environment-variable fallback. Users will commit real keys. Load from `os.environ` with a fallback.
8. **No error recovery in streaming** — If the first LLM pass raises `RuntimeError`, the exception propagates to the frontend with no partial-result fallback. Catch the error after Pass 1 and return whatever text is available.

## Pipeline aggression mismatch in LLM stage

In `pipeline.py` lines 109 and 164, the `LLMRewriter` is instantiated with:

```python
stage = LLMRewriter(
    aggression=self.stage_config.get("lexical_aggression", 0.5),
    ...
)
```

There is no `llm_aggression` key in the `INTENSITY_PROFILES` dict in `config.py`. The LLM stage's aggression parameter controls both temperature scaling and which prompt variant is used (the threshold checks at aggression 0.4 and 0.7 in `_get_pass1_prompt`). Because lexical and LLM aggression should differ — you may want aggressive vocabulary injection but conservative rewriting — this coupling means the LLM stage can't be tuned independently.

## Streaming display and the Pass 2 reset

In `pipeline.py` `process_stream`, when the `---PASS2---` sentinel arrives the method resets `llm_chunks = []`. The frontend in `app.py` collects chunks and renders them cumulatively:

```python
for chunk in pipeline.process_stream(input_text):
    if chunk == "\n\n__FINAL__\n\n":
        final_override = ""
        continue
    if final_override is not None:
        final_override = chunk
        continue
    collected_chunks.append(chunk)
    stream_container.markdown(''.join(collected_chunks))
```

But the `---PASS2---` sentinel is only consumed inside `pipeline.py`'s `process_stream` — it never reaches `app.py`. This means the frontend accumulates *all* yielded chunks including both Pass 1 and Pass 2 text concatenated. The displayed streaming output is Pass 1 + Pass 2 concatenated, and the variable `current_text` inside the pipeline after streaming is only Pass 2 (because the internal list was reset). If post-processing then runs and produces a different string, the `__FINAL__` sentinel triggers and the UI replaces the concatenated mess with the correct post-processed text. If post-processing makes no changes, the UI keeps the concatenated Pass 1 + Pass 2 as the final output, which is wrong.

## Non-determinism across all NLP stages

`stage_structural.py`, `stage_lexical.py`, `stage_perplexity.py`, and `stage_postprocess.py` all call `random.random()` and `random.choice()` using the global random state. There is no seed parameter on any stage constructor or on the pipeline. This means:
- A user cannot reproduce a result they liked.
- Automated tests are flaky.
- Two runs at identical settings produce different outputs, eroding trust for an academic user who wants to iterate.

## Lexical replacement regex hazard

In `stage_lexical.py` `_replace_target_words`:

```python
for word, replacements in self.target_words.items():
    pattern = re.compile(r'\b' + word + r'\b', re.IGNORECASE)
    matches = pattern.finditer(text)
    for match in matches:
        if random.random() < self.aggression * 0.7:
            replacement = random.choice(replacements)
            text = text[:match.start()] + replacement + text[match.end():]
            break
```

The `break` after the first replacement prevents cascading corruption, but it also means the function can replace at most one occurrence of each target word per call — regardless of text length or aggression setting. If a 5000-word essay uses "significant" 12 times, only one gets replaced. At aggression 0.9, the user expects aggressive substitution but gets one swap per word type.

Additionally, if the replacement has a different length than the original (e.g., "significant" → "notable" or "considerable"), any subsequent regex match positions computed by `finditer` on the original string would be invalid. The `break` masks this bug, but if it were removed to fix the single-replacement limit, the position corruption would produce garbled output.

## AI risk score provides circular validation

`compute_ai_risk_score` in `text_analysis.py` measures four factors: sentence length variance, transition word density, type-token ratio, and perplexity uniformity. The pipeline's stages are specifically designed to increase sentence length variance (Stage 1), reduce transition words (Stage 5), increase TTR (Stage 2), and inject perplexity variance (Stage 4). The score will mechanically decrease after processing regardless of whether the text actually evades Turnitin. This gives the user a progress bar that always goes in the right direction, which is epistemically dishonest — the user's original complaint was that Quillbot reported success but Turnitin still flagged the text. This metric is likely to reproduce the same false-confidence pattern.

## Credential management

`config.py` hardcodes `API_KEY = "insert your api key here"` and `BASE_URL = "https://api.freemodel.dev"`. There is no `os.environ.get("API_KEY", ...)` pattern. Users editing this file will commit their real API key to version control. The README instructs users to "Edit `humanizer/config.py` to set your API key" rather than using environment variables or a `.env` file (which should be gitignored).

## No partial-result recovery on LLM failure

`LLMRewriter._llm_pass_stream` raises `RuntimeError` if the HTTP request fails. In `process_stream`, if Pass 1 succeeds but Pass 2's request times out, the exception kills the entire pipeline. The pre-LLM stages (structural + lexical) already ran and produced improved text, but that intermediate result is lost. A try/except around Pass 2 that falls back to Pass 1 output would preserve partial progress.
