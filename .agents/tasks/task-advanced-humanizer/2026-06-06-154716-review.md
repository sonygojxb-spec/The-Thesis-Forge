# Multi-stage humanization pipeline (v2)

The eight issues from the v1 review have been addressed: `llm_aggression` key added to all intensity profiles, streaming redesigned so only Pass 2 yields to the frontend, post-processing uses a clean `__POSTPROCESSED__` sentinel, a `seed` parameter threads through all NLP stages, lexical replacements now collect first and apply in reverse order, an AI risk disclaimer exists in the UI, the API key loads from `os.environ.get("THESIS_FORGE_API_KEY", ...)`, and LLM failures fall back gracefully to pre-LLM text.

**Watch for:** The `__POSTPROCESSED__` sentinel protocol in `process_stream` has a logic flaw — the frontend collects only the *last* chunk after the sentinel, but `process_stream` yields *one* chunk containing the entire final text, so it works by coincidence rather than contract. If any future post-LLM stage yields additional chunks, the frontend will silently drop intermediate ones (**confirmed**). All NLP stages call `random.seed(self.seed)` at the top of `process()`, resetting the global random state — this creates cross-stage interference where stage ordering changes affect reproducibility, and any concurrent pipeline runs (e.g., two Streamlit sessions) corrupt each other's random sequences (**confirmed**). The perplexity stage's `_complexify_sentence` can produce malformed text by inserting parenthetical phrases that break punctuation rules (double commas, dashes adjacent to existing punctuation) (**confirmed**).

## High-level view

The v1 fix pass addressed all flagged issues and the pipeline is structurally sound. The streaming redesign is the most significant improvement — Pass 1 now runs internally with chunks collected but not yielded, and only Pass 2 output streams to the user. The fallback chain (Pass 2 fails → use Pass 1; Pass 1 fails → use pre-LLM text) is well-structured with appropriate try/except boundaries.

The seed-based reproducibility fix uses the global `random.seed()` call rather than per-stage `random.Random` instances. This means stages are only reproducible when run in the same order with the same preceding stages, and concurrent invocations will corrupt each other. For a single-user Streamlit app this is unlikely to cause problems in practice, but the reproducibility guarantee is weaker than it appears.

The lexical regex fix correctly collects all replacements first and applies them in reverse positional order, eliminating the position-corruption bug. However, the function now replaces multiple occurrences of the same word — which is the desired behavior — but the aggression gating logic applies at two levels (per-word skip and per-match skip), creating a compound probability that makes effective replacement rates lower than the aggression value suggests.

The post-processing sentinel swap from `__FINAL__` to `__POSTPROCESSED__` is cleaner semantically but the protocol still relies on a single-chunk assumption in `app.py` that isn't enforced by the pipeline's interface contract.

## Global random state cross-contamination

Every NLP stage follows the same pattern:

```python
def process(self, text):
    if self.seed is not None:
        random.seed(self.seed)
    ...
```

This resets the *module-level* random state. In a Streamlit deployment, two users hitting "Humanize Text" simultaneously will interleave `random.seed()` and `random.random()` calls across threads (Streamlit runs each session in a separate thread). The result is that neither session gets reproducible output — the seed guarantee breaks under concurrency.

The fix is straightforward: each stage should instantiate `self.rng = random.Random(seed)` and use `self.rng.random()` / `self.rng.choice()` throughout. This isolates each stage instance's state without affecting the global generator.

## Sentinel protocol fragility in streaming

In `pipeline.py`, `process_stream` yields:
```python
yield "\n\n__POSTPROCESSED__\n\n"
yield post_text
```

In `app.py`, the consumer:
```python
for chunk in pipeline.process_stream(input_text):
    if chunk == "\n\n__POSTPROCESSED__\n\n":
        postprocessed_text = ""
        continue
    if postprocessed_text is not None:
        postprocessed_text = chunk
        continue
```

The line `postprocessed_text = chunk` (assignment, not append) means if multiple chunks arrive after the sentinel, only the last one is kept. Currently this works because `process_stream` yields exactly one chunk after the sentinel. But the interface doesn't document or enforce this constraint — it's an implicit protocol that will break silently if the post-processing path evolves to yield incrementally.

## Perplexity stage punctuation artifacts

`_complexify_sentence` in `stage_perplexity.py`:

```python
if random.random() < 0.6:
    insertion = random.choice(self.parentheticals)
    insert_pos = random.randint(3, min(8, len(words) - 2))
    if random.random() < 0.5:
        words.insert(insert_pos, f"- {insertion} -")
    else:
        if not words[insert_pos - 1].endswith(','):
            words[insert_pos - 1] = words[insert_pos - 1] + ','
        words.insert(insert_pos, f"{insertion},")
```

This doesn't check whether `words[insert_pos - 1]` already ends with punctuation other than a comma (e.g., a semicolon, period, or another dash). Inserting `"- at least in part -"` after a word that ends with `;` produces `word; - at least in part -`. The comma path also doesn't check whether the word at `insert_pos` starts with punctuation or a conjunction that would make the insertion grammatically awkward.

The `_final_cleanup` in the postprocess stage catches some of these (double spaces, space-before-punctuation), but not all — specifically, it won't fix double commas (`,,`) or dash-adjacent-to-semicolon patterns.
